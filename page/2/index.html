paypal对接流程

概述

国内的支付接口相对一致，文档也很丰富。相比来说，paypal的接口复杂很多，并且文档大部分都是英文的。这次对接paypal基本的调研用了将近两天，才搞出个demo。简单记录下流程。

帮助地址

  • paypal php sdk on github 地址
  • 开发者中心 地址

流程

  1. 注册普通账号。
  2. 升级为企业帐号。（国内版和国际版有区别，跟客服电话沟通，确认已经取消个人的高级账号，只保留企业帐号，但是可以通融，个人申请企业版，客服不会禁止。这一点，国外相对宽松，个人帐号能正常升级。国内一般几个小时就能正常使用） [地址：个人首页，左侧有帐号升级入口]
  3. 生成api认证。完成后会提供 live版的帐号、密码和签名。 地址
  4. 进入开发者中心 地址
  5. 添加应用，成功后，会提供client id和client secret，第三方支付基于此。 地址
  6. paypal很人性化，提供了一套沙盒，可以使用live帐号，申请多个测试的沙盒帐号（包括用户和商户），金额随便充。地址
  7. 参考github samples。创建demo，使用沙盒帐号测试支付和付款的完整流程。地址：根据需要选择demo
  8. 对接网站，使用创建订单时paypal返回的 pay-id 来绑定 order-id，用于支付成功的回调更新order信息。

备注

  • 创建订单后的请求、响应信息
  • 支付成功后返回信息[get]

在线网页转PDF接口服务

概述

以前做过一些PDF生成的程序，主要是tcpdf，但是排版很复杂，需要手动计算位置、大小、贴图。

方案

Html/css排版是非常方便快捷的，所以就考虑能不能将网页转换成PDF。由此衍生出两种方案

  1. html 转 图片，使用tcpdf（或者其他语言的其他库）创建pdf，每一页使用一个图片。优点：基本保持网页原貌，每一页大小好控制。缺点：文件偏大，文字不能复制。

  2. html直接转PDF。开始找了很多文章，没有解决。期间，js方案倒是有，但是，前端本身就不是干这个事情的东西，放弃。后来学习其他技术，无意中找到了一套很好的解决方案。

结果

做了一个独立接口，只要有授权，传递一个url，即可生成。做成服务的话，还有很多可以完善，比如支持文件上传、在线编辑pdf排版、文字等。

5月2D-h5游戏开发月

概述

更新文章很慢了，其实很多要记录的东西，碎片化的不止是知识还有各种事项。

这个5月基本上以2D游戏开发为主，以前collie、coocs2d js、egret（关注从0发展至今）都做过，这几年发展比较快，再次关注游戏开发，发现webGL效率非常高了，在微信里表现也非常好。

这次游戏开发重新筛选了引擎，最终选择phaser，非常优秀的一款，支持typescript。主要特点：轻便灵活，可定制性高。

游戏开发要点总结

游戏开发涉及的方面非常的广，2D相对来说还算简单。此文单聊2D。

工具

  • 图片合并
  • 骨骼动画
  • 字体合并
  • 瓦片地图
  • 物理引擎编辑
  • 编辑器（目前vim）
  • gif分割工具

游戏引擎

  • 场景
  • 精灵（多种加载方式，图片、拼图）
  • UI
  • 动画（补间、逐帧）
  • 物理引擎

开发流程

  • 分解素材
  • 搭建框架
    • 代码方面对 缓存、游戏对象(场景、精灵、ui)管理、引擎方法优化
    • 服务层封装
  • 逐个场景进行开发，首先静态场景，然后对游戏逻辑开发，最后对接服务层

总结

伴随游戏容器的效率提升和对webGL的支持越来越好，h5游戏开发会比较常态化，但对代码保护需要专门的工具。素材的话，基本是透明的，很难处理，跟端游的境遇差不多，淘宝几块钱就能买一套游戏的全部素材。

另外，对游戏后端的开发部署，除去本身对nodejs这种东西的嫌弃，这个算是不错的选择，有很多分布式的架构设计方案。

docker 使用记录

Docker 个人基本理解

  • 镜像 静态的资源文件，用于容器中动态加载
  • 容器 动态的文件，用于承载镜像。类似操作系统之于应用软件
  • pull commit 之类，将镜像通过类似git的方式管理起来，方便分发和版本管理
  • 能做什么？一致化的模块功能，方便的组合，跨平台部署

Docker commands

  • systemctl start docker
  • docker pull // 获取镜像， 可以先通过git search 查找
  • docker images // 获取所有镜像
  • docker run -v hostpath:vmpath -v hostpath2:vmpath2 –privileged -p 80:8080 –name renameTest centos:latest -it /sbin/init
  • -v和volume配置有不同。主要是文件夹的共享范围。宿主和容器、容器和容器之间是否可以同步修改。参考此文
  • -it
  • –privileged 部分命令需要宿主root权限
  • -p 和宿主端口映射
  • –name 起个小名
  • docker commit // 可以将容器打包成镜像
  • docker ps -a // 所有容器
  • dockerfile配置详解，参考该网址

2017年一个私有项目计划

立意

以前一直想做个利用手机传感器控制应用或者游戏的东西，最近对构思基本完善，准备用一年的业余时间玩玩。

架构

主要由服务端、采集端、客户端、管理端组成。

业务图

技术需求

  • java/android oc/ios
  • cpp/go
  • html/css/js/web
  • cocos2d/three

开发安排

第一期（3个月）

  • andorid采集数据、连接服务端发送数据
  • 主要完成服务端数据接收、转发数据到客户端（主要TCP/UDP/WS协议）
  • 完成数据端 传感器数据解析 （比如三轴传感器的数据解析成）
  • 基础客户端（基于web或者其他协议的数据展示）

第二期

  • ios采集数据端
  • 采集端可以选择控制的客户端
  • 服务端基本隔离出解析（算法模块）、数据收发模块、转发模块
  • 基于数据库的管理配置（服务端配置-解决连接数等控制；采集端配置-解决权限实时控制）
  • 完成一个基本的游戏，实现控制

第三期

  • 采集端优化（ui、连接稳定性、本地缓存）
  • 服务端优化（数据收发的队列管理系统）

第四期

  • 解决部分并发问题
  • 【暂无能力】分布式部署服务端
  • 传感器数据解析算法（有非常完善的解析，包括多种动作）
  • 客户端实现应用平台化，制作客户端SDK包

备注

  • 项目可能会停留在第一期完成。
  • 困难不少，IOS考虑二期拉入其他开发。
  • 根据发展，代码成熟考虑开源。

进度更新

  • 2017-04-21 完成单机服务端、android采集端。工作很忙，后面用threejs做一个使用终端的demo。
  • 2018-07-04 可惜，在最近一年被各种事项严重压缩睡眠时间后，该计划停留在了二期… :( 默哀1秒钟，心疼缺觉的少年

新计划

生活上，摇号估计没戏，一定要来个外地牌照的小车了。
