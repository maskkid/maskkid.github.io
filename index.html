docker总结

背景

很看好一些可预见发展强势的业务场景下的微服务架构，从硬件到软件，从技术架构到业务隔离，彻底解耦，并且从软硬两层制定了严格的耦合场景标准。

之前一直用PHP做项目，语言本身可选择性很低，因为需要配合nginx和php-fpm等，基于虚拟化的微服务架构，显得特别复杂和臃肿。有一段时间，想彻底转到golang（主要是早期七牛全go架构和用golang做过一个跨平台的监控系统, 玩的爽到不要不要的）,另外一个选择就是nodejs（这货，从一出来，就不看好，之前一直基于v8解析，另外那时候js的工程化生态非常弱）,但在天津这个国际化大都市的互联网环境里，go的生存空间太小了，也只能当工具。

这几天，正好有时间，抛开语言，重新从架构垂直分层的底层开始准备，大概跟踪了docker最新的一些生态玩法。

基础

  • docker image
  • docker container
  • commit/push/save
  • DockerFile
  • network
  • link
  • volume

  • docker-compose
    • services
    • links

      • context

  • 一个docker颗粒度

  • docker基本管理，变更、组合(docker file, compose)
  • 分布式docker管理工具**(k8s)

  • 大量docker容器内软件的数据规划没有标准，比如日志、文件、数据等等
  • 问题定位、跟踪、调试工具

关于基于gitX的workflow

概要

产品一般会经过需求(播种)、研发(孕育)、上线(出生)几个过程，整个流程涉及到多个独立个体（团队），这期间的通信和日志得非常重要，因为这对于一个产品、一个公司都是一套动态标准化的实例，有助于适应团队的标准化工作流的产生。这里，有一个比较特殊的团体，就是开发团体，尤其是互联网企业，而git这种代码管理工具也逐渐被扩展成以研发为中心，向上下游扩展的工作流工具。

目前github,gitlab,gogs等等基本的都包含issue（需求、测试），codes（研发，又包含并行、串行代码变更的管理方式），甚至涉及项目管理、基本的成本统计等。这里，简单记录下关于代码阶段的gitX workflow

选择

主要根据这几点因素考虑选型

  • 项目开发周期
  • 项目生命周期
  • 项目开发所处阶段
  • 团队人员规模
  • 团队技能基础

基础分支建议：

master(发版)

develop(开发)

    feature-xxx 

test(需不需要测试)

hotfix

总之，原则就是 “适度”，多个分支之间就是互相守候的关系，根源在于并行，只要团队和项目接受，可以从简到繁。

为什么选择react

前段时间总结的一些开发问题

这是前段时间在一套不认可的前端方案下，后端几十号人完全没有规范的情况下，总结了一个问题和建议的文档。

前端部分

  1. 框架底层库不稳定,尤其是自建库部分,目前经常需要变动,导致页面来回改动. 建议: 封一层,内部完全只用antd.
  2. 目前写一个页面创建需要做以下几件事:
    • 添加路由配置(改写loadfile和index两个文件) 2
    • 分析页面数据结构,创建数据结构体(前端叫法 interface) 1
    • 分析页面,隔离编写组件/容器/页面三部分 2-3
    • 编写redux以及数据服务部分,涉及以下几个目录和文件的创建:
      • actions/模块名/页面文件 1
      • reducers/模块名/页面文件,创建完成后,改写同目录下index文件配置,然后再改写reducers目录下的index文件 3
      • api 1
      • mock编写(mock具体文件编写/db.js/routes.js) 3

以上过程,纯手工编写,非常繁琐,建议:尽量把路由/api/mock部分提前配置,自动生成(依赖后端同事,尽快出全部的接口定义)

接口部分

  1. 文档不统一,生成的文档每个模块接口都不一样
  2. 没有接口规范,可以参考币别管理接口
  3. 接口容错和提醒很暴力,最起码应该区分一下业务错误是数据格式问题,还是数据项类型问题
  4. 接口交付前,一定要测试,并且有测试通过的原始数据格式和测试结果记录

接口要么没有定义;有定义的,没字段注释;有字段注释的,调不通;调通了,又要改

对接例子eq.
企业信息接口 查询/添加 一周
摊销 一周
财税设置 一周
币别管理 四个接口 一下午
